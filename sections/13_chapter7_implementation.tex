\chapter{Implementation}

\section{Module-wise Implementation}
Implementation is organized into independent modules:
\begin{itemize}
\item \textbf{UI Module (`voteguard/app.py`, `src/ui/*`):} voter flow screens, validation dialogs, and cast confirmation.
\item \textbf{Core Module (`voteguard/core/*`):} domain entities, state machine, and use cases (cast, tally, verify).
\item \textbf{Storage Module (`adapters/storage_fernet_hashchain.py`):} encrypted append-only ballot ledger with hash chaining.
\item \textbf{Audit Module (`adapters/audit_log_hashchain.py`):} tamper-evident event trace.
\item \textbf{Device Module (`adapters/devices_mock.py`, `src/hardware/*`):} hardware abstraction and simulation.
\item \textbf{Blockchain Module (`adapters/chain_simulated.py`, `src/blockchain/*`):} chain-anchoring pathway and prototype integration hooks.
\end{itemize}

\section{Code Snippets and Explanation}
Representative implementation pattern:
\begin{verbatim}
def cast_vote(voter_hash, candidate_id):
    if cast_registry.exists(voter_hash):
        raise DuplicateVoteError()

    payload = {
        "candidate_id": candidate_id,
        "timestamp": now_iso(),
        "station_id": station_id,
    }
    encrypted = fernet.encrypt(json.dumps(payload).encode("utf-8"))
    record = ballot_ledger.append_with_hashchain(encrypted)
    audit_ledger.append_event("vote_cast", record.hash)
    cast_registry.add(voter_hash)
    return record
\end{verbatim}

\section{Database Implementation}

\subsection{Schema and Structure}
A relational database is not mandatory for the prototype. Persistent state is maintained using structured JSON ledgers and registry files.

\subsection{SQL Table Structures}
SQL is not used as primary persistence in this build. Equivalent logical schemas are represented as ledger records and registry entries.
