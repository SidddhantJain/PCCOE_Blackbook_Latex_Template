\chapter{Implementation}

\section{Module-wise Implementation}
Implementation is organized into independent modules:
The UI module in voteguard/app.py and src/ui/* manages voter flow screens, validation dialogs, and cast confirmation. The core module in voteguard/core/* contains the domain entities, the state machine, and the use cases for casting, tallying, and verification. The storage module in adapters/storage_fernet_hashchain.py provides the encrypted append-only ballot ledger with hash chaining, while the audit module in adapters/audit_log_hashchain.py maintains the tamper-evident event trace. The device module in adapters/devices_mock.py and src/hardware/* handles hardware abstraction and simulation, and the blockchain module in adapters/chain_simulated.py and src/blockchain/* provides the chain-anchoring pathway and prototype integration hooks.

\section{Code Snippets and Explanation}
Representative implementation pattern:
\begin{verbatim}
def cast_vote(voter_hash, candidate_id):
    if cast_registry.exists(voter_hash):
        raise DuplicateVoteError()

    payload = {
        "candidate_id": candidate_id,
        "timestamp": now_iso(),
        "station_id": station_id,
    }
    encrypted = fernet.encrypt(json.dumps(payload).encode("utf-8"))
    record = ballot_ledger.append_with_hashchain(encrypted)
    audit_ledger.append_event("vote_cast", record.hash)
    cast_registry.add(voter_hash)
    return record
\end{verbatim}

\section{Database Implementation}

\subsection{Schema and Structure}
A relational database is not mandatory for the prototype. Persistent state is maintained using structured JSON ledgers and registry files so that the EVM workflow remains lightweight and portable.

\subsection{SQL Table Structures}
SQL is not used as the primary persistence layer in this build. Equivalent logical schemas are represented as ledger records and registry entries.
