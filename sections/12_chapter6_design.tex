\chapter{Design}

\section{Use Case Diagram}
Primary actors are Voter, Election Officer, and System Auditor. Key use cases include voter enrollment validation, biometric capture, cast vote, verify ledger integrity, and generate tally.

Use case description summary:
\begin{itemize}
\item \textbf{Preconditions:} voting session initialized, candidate list available, device stack active or in simulation mode.
\item \textbf{Main flow:} identifier entry $\rightarrow$ biometric capture $\rightarrow$ vote cast $\rightarrow$ encrypted storage.
\item \textbf{Alternative flow:} hardware unavailable, fallback simulation with audit flags.
\item \textbf{Postconditions:} vote recorded, cast-registry updated, audit event persisted.
\end{itemize}

\section{Class Diagram}
The class model centers on clean architecture boundaries:
\begin{itemize}
\item Core domain and use cases (`domain.py`, `usecases.py`, `state_machine.py`).
\item Port interfaces (`ports.py`) for storage, audit, chain anchoring, and devices.
\item Adapter implementations (`storage_fernet_hashchain.py`, `audit_log_hashchain.py`, `devices_mock.py`, `chain_simulated.py`).
\end{itemize}

\section{Sequence Diagram}
Critical sequence: Voter UI invokes CastVote use case, use case calls storage port to append encrypted record, then audit port logs event, followed by UI confirmation and transition to start state.

\section{Activity Diagram}
Activity flow includes decision gates for identifier validity, camera/device readiness, duplicate vote checks, and storage exception handling paths.

\section{Entity-Relationship (ER) Diagram}
The prototype uses file-ledger entities instead of relational tables:
\begin{itemize}
\item \textbf{BallotRecord:} sequence number, encrypted payload, previous hash, current hash, timestamp.
\item \textbf{AuditRecord:} event type, metadata, previous hash, current hash, timestamp.
\item \textbf{CastRegistry:} salted identifier hash, cast timestamp, session metadata.
\end{itemize}

\section{Component Diagram}
Components are split into UI, Core Use Cases, Storage Adapter, Audit Adapter, Device Adapter, and Optional Chain Adapter. Dependencies are inward toward core interfaces only.

\section{Deployment Diagram}
Deployment is designed for polling-station edge execution:
\begin{itemize}
\item Windows/Edge compute unit running UI and core services.
\item Camera and biometric peripherals connected locally.
\item Local encrypted storage for ballots and audit logs.
\item Optional network uplink for chain anchoring and centralized monitoring.
\end{itemize}
