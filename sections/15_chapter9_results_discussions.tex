\chapter{Results and Discussions}

\section{Experimental Setup and Testing}
The prototype was evaluated in a controlled development setup using simulation-first execution and optional camera input. The evaluation considered correctness of cast workflow, tamper-evident storage behavior, and script-based tally/verification.

Setup characteristics:
The setup used a desktop or laptop Windows environment with a Python virtual environment, repository-provided scripts for simulate/cast/tally/verify cycles, sample vote datasets generated through simulation scripts, and optional camera-device runs for the biometric stage of the UI.

\section{Development Environment and Tools}

\subsection{Programming Languages, Frameworks, and Libraries}
Technologies used in the project:
The project uses Python as the main implementation language, with C++ and Rust present only as prototype-level stubs. The interface is built with PyQt5, the backend follows clean-architecture Python modules in voteguard/core and the adapters layer, and the dependencies include cryptography, pytest, OpenCV for optional camera support, ONNXRuntime for optional analytics, and web3 as a placeholder pathway.

\subsection{Integrated Development Environment (IDE)}
Primary IDE and tooling: Visual Studio Code, PowerShell, and Python CLI workflow.

\subsection{Databases and APIs}
List:
The project uses JSON-based encrypted ballot and audit ledgers that are file-backed rather than relational. Third-party APIs and services appear only as optional blockchain and web3 integration placeholders, while external integrations cover camera hardware and biometric adapter interfaces.

\subsection{Development Tools}
Include version control, testing frameworks, build tools, and other utilities:
Version control is handled through Git and GitHub. Testing relies on pytest and custom verification scripts, build setup uses Python packaging and requirements-based installation, documentation is maintained through Markdown files and diagram XML or Draw.io assets, and deployment is performed using scripted local execution such as run_app.py and run_count_app.py.

\subsection{Functional Results}
Present results demonstrating successful feature implementation through:
The system demonstrated end-to-end voting flow execution across the UI stages, encrypted ballot ledger creation and append behavior at cast time, consistent outputs from the tally and ledger verification scripts, and successful fallback behavior for optional components.

\section{Analysis and Interpretation}
Interpretation of outcomes:
The results show that security and traceability goals are met at prototype level through encrypted and chained records. The modular architecture reduced integration friction and simplified testing, while optional ML overlays improved observability without being used for vote eligibility decisions. The simulation-first design also ensured operability in constrained or hardware-limited setups.
